❯ Explain why we split on |||| instead of new lines.                            

⏺ The Shortcut's "Split Text" action is dumb — it splits on a literal fixed
  string. If we split on newlines, the first item would be the title (correct),
  but getting "everything after the first line" as a single string isn't
  possible in Shortcuts. There's no "get items 2 through end and rejoin" action.

  |||| gives us exactly two parts — title and body — so First Item and Last Item
   each grab exactly what we want, even if the body contains many lines or blank
   lines.

❯ But I want to skip the first line (Page Number) and use the second as the     
title. Here's exactly what one of the text files looks like:                    
Page 24                                                                         
# 3/22                                                                          
-killed it today all day. I stayed regulated& connected w/ the girls. We        
returned from our road trip today. Keep it up, dude.                            
⏺ Python handles the line 1/2/body split — that's all done in code before the
  Shortcut is ever called. By the time the Shortcut sees anything, Python has
  already:

  1. Discarded Page 24
  2. Stripped #  from # 3/22 → title = 3/22
  3. Joined remaining lines → body = -killed it today all day...
  4. Built the payload: 3/22||||-killed it today all day...

  The |||| exists purely because the body itself contains newlines. If we passed
   3/22\n-killed it today... and split on newlines in the Shortcut, Last Item
  would only return the last line of the body, not all of it.

  Also worth updating the plan — Python should strip the #  prefix from the
  title line. I'll add that:

❯ It appears the Shortcut/URL Schema for Journal is only available on iOS and

⏺ The architecture needs to change. A few real options:

  Option A: Day One CLI
  Day One has a proper macOS CLI (dayone2). Entries sync to iPhone via iCloud.
  Straightforward drop-in replacement — just swap the journal step.
  dayone2 new "entry text" --title "3/22"
  Downside: Day One is a paid app ($35/yr).

  Option B: iCloud file drop → iOS Shortcut
  Mac writes the entry as a .txt file to an iCloud Drive folder. An iOS Shortcut
   automation watches that folder and creates a Journal entry when new files
  appear. Preserves Apple Journal on iPhone, but the Mac is no longer the
  execution host for the Journal step — just a file producer. Slightly fragile
  (iOS automation can be flaky).

  Option C: Apple Notes via AppleScript
  Apple Notes is fully scriptable on macOS and syncs to iPhone natively. Not
  quite the same as Journal, but zero cost and reliable automation:
  osascript -e 'tell application "Notes" to make new note with properties 
  {name:"3/22", body:"..."}'

  Option D: Markdown files in Obsidian
  Write .md files to an Obsidian vault in iCloud. Not Apple Journal, but highly
  automatable and sync is solid.

  My recommendation depends on one question: are you attached to Apple Journal 
  specifically, or is the goal just a private, synced journal on your iPhone? If
   it's the latter, Day One or Notes are much cleaner paths.

❯ How do I grant access to my gmail if I have 2fa enabled?                      

⏺ You need to create a Gmail App Password — a one-time 16-character password
  that lets the daemon authenticate via IMAP without your real password or 2FA
  prompt.

  1. Go to myaccount.google.com
  2. Search for "App Passwords" in the search bar (it's buried otherwise)
  3. Give it a name like kindle-journal
  4. Click Create — Google shows you a 16-character password once
  5. Copy it immediately and put it in your .env file:
  IMAP_USERNAME=you@gmail.com
  IMAP_PASSWORD=xxxx xxxx xxxx xxxx
  5. (spaces in the password are fine — or remove them, both work)

  Prerequisite: App Passwords only appear if your Google account has 2FA
  enabled, which you already have. If you don't see the option, search for it —
  Google has moved it around and sometimes hides it depending on account type.
